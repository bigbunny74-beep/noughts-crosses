CHANGE 1: Firework sound replaced with realistic multi-layer audio
  The old SFX.firework was a single sawtooth oscillator at 100Hz followed
  by a random sine tone — indistinguishable from a generic game bleep.

  Replaced with playFireworkSFX(big), a four-layer Web Audio API synthesis
  chain that models real pyrotechnic sound physically:

  LAYER 1 — LAUNCH WHISTLE (0ms–180ms)
    A sine oscillator sweeps from 180Hz to 820Hz over 180ms using
    linearRampToValueAtTime. This mimics the aerodynamic whine of a
    firework shell rising through the air. Quiet volume (22% of base vol)
    so it supports but doesn't dominate the burst.

  LAYER 2 — BURST CRACK (fires at 170ms)
    White noise (AudioBuffer filled with Math.random()*2-1, 320ms long)
    routed through a BiquadFilter highpass at 900Hz (Q=0.8). The highpass
    removes the low-frequency rumble, leaving only the sharp transient
    crack/pop of a real explosion. Attack is 2ms (linearRamp from near-zero
    to full), decay is 280ms exponential — identical shape to a real burst.
    This is the most important layer; oscillators cannot replicate it.

  LAYER 3 — SPARKLE TAIL (fires at 190ms)
    A second white noise buffer (500ms) routed through a bandpass filter
    centred at 2200Hz (Q=1.4). This frequency band is the hiss, sizzle and
    crackle of falling burning sparks — the sound that lingers after the
    initial burst and gives fireworks their sustained texture. Fades over
    450ms exponentially.

  LAYER 4 — BASS THUD (big bursts only, fires at 170ms)
    A 65Hz sine oscillator with an 8ms linear attack and 100ms exponential
    decay. This adds the physical low-frequency impact that large celebration
    fireworks have — the "punch" you feel in your chest. Only fires when
    big=true (launchVictoryCelebration big=true), keeping normal fireworks
    lighter and the big win moments impactful by contrast.

ADVISORY — Fireworks, Sound and Music
======================================

FIREWORKS — making them look more real (without changing the particle system)

The current fireworks use circular particles launched from random positions.
To make them look more realistic, three things matter most:

1. SHELL TRAIL: Real fireworks launch from a point with a bright white streak
   before the burst. Add a short-lived white particle that travels upward fast
   before spawning the burst — this is just one extra particle with a steep
   upward velocity that fades in 200ms.

2. GRAVITY + AIR RESISTANCE on particles: Currently particles likely move at
   constant velocity. Real burst particles arc downward and slow down. Adding
   a small gravity constant (vy += 0.08 per frame) and a drag multiplier
   (vx *= 0.97, vy *= 0.97) per frame makes the burst look organic.

3. TWINKLE/FADE: Instead of linear fade, real sparks flicker — brightness
   oscillates slightly as they fade. A sin(particle.life * 20) * 0.15
   added to opacity gives a convincing sparkle with zero extra objects.

These three changes together — shell trail, gravity, twinkle fade — would
transform the fireworks from "coloured dots expanding outward" to something
that reads as real pyrotechnics on screen.

SOUND — making it better

The current audio uses a single Web Audio API oscillator chain. Improvements:

1. BETTER LAUNCH SOUND: A white-noise burst for 80ms followed by a short
   ascending tone (100Hz → 400Hz in 120ms) mimics the launch whistle.
   White noise = buffer filled with Math.random()*2-1 run through a BiquadFilter.

2. BETTER BURST SOUND: Instead of a sawtooth tone, the burst should be a
   noise burst (wide bandwidth) with a fast attack (2ms) and slower decay
   (300ms), filtered with a high-pass filter around 800Hz to remove the
   low rumble and keep the crack/sparkle. Add a second noise burst at 20ms
   delay at lower volume for the "echo" effect of a real explosion in open air.

3. MOVE SOUNDS: The current X and O placement tones work well but adding
   very subtle pitch variation (±5% random) each play makes repeated moves
   feel less mechanical.

MUSIC — background music

Yes, it is absolutely possible using Web Audio API. There are two practical
approaches for a game this size:

APPROACH A — Procedural generative music (recommended)
  Generate looping music entirely in JavaScript using oscillators, timing,
  and note tables. No audio files, no loading, no licensing.

  Example — a gentle, looping waltz-time melody appropriate for Hearty:
  A pentatonic scale (C D E G A) in 3/4 time at ~120 BPM generates something
  warm and non-aggressive. The melody is a small array of note frequencies
  played on a soft sine/triangle oscillator with a slow attack (80ms) and
  release (200ms). A simple bass note (root of each chord) plays on beat 1.
  A soft pad (two detuned oscillators, 4 cents apart) plays sustained chords.
  Total: ~60 lines of JS, no external dependencies, loops seamlessly.

  The music would mute automatically when the player makes a move (using the
  existing _soundOn flag) and fade back in after 2 seconds of inactivity,
  keeping it from competing with the SFX.

APPROACH B — Embedded base64 audio (simpler but larger file)
  A short looping audio file (8–16 bar loop, ~15 seconds, MP3 at 64kbps
  ≈ 120KB) encoded as base64 and embedded directly in the HTML. An
  <audio loop> element handles all the playback.
  Pros: richer sound, easier to control feel/mood.
  Cons: adds ~160KB to the file, requires creating the audio asset.

RECOMMENDATION:
  Approach A (generative) is the better fit for this game — it stays as a
  single self-contained HTML file, the music can adapt to theme (Halloween
  theme gets a minor-key melody, Spring gets a brighter major key), and it
  costs nothing in file size. If you want me to implement it, say the word
  and I will build the full procedural music engine into the game.

CHANGE 1: Persistent encouragement memory — Hearty gets warmer the more the player struggles
  This is the single biggest personality change to make Hearty feel like a real caring
  AI character rather than a static mascot.

  HOW IT WORKS:
  A lossStreak counter in state tracks how many consecutive times Hearty has beaten
  the player in vs-Hearty mode. The counter increments on every Hearty win and resets
  to 0 on a player win, a draw, or a mode switch. It is completely separate from the
  existing winStreak system.

  FOUR TIERS OF RESPONSE (lossStreak thresholds):
  Tier 1 — 1st loss:
    Hearty combines a gentle apologetic win message with immediate warm encouragement.
    Bubble animation: standard bubbleIn (gentle rise).
    e.g. "I scraped through that one! You were so close...
         One game closer to cracking it! I believe in you!"

  Tier 2 — 2nd and 3rd loss:
    Hearty drops the win message entirely and leads with a specific actionable tip.
    Bubble animation: bubbleEncourage — a soft overshoot bounce, more energetic than normal,
    signalling Hearty is leaning in with genuine enthusiasm.
    e.g. "Try grabbing the centre square first — it opens up so many more paths! 💡"

  Tier 3 — 4th and 5th loss:
    Hearty is now visibly concerned and rooting hard. Messages are longer, more personal,
    and give away genuine strategy secrets. Hearty is essentially coaching the player.
    Bubble animation: bubbleCheer — a double-bounce with larger overshoot, as if Hearty
    is physically jumping up to get the player's attention with encouragement.
    e.g. "I'm rooting for you SO much right now! Here's my secret: I always block the
         move that threatens two lines. So set up threats in two directions and I can't
         stop both! 💪"

  Tier 4 — 6+ losses:
    Full cheerleader mode. Hearty is practically beside themselves. Messages are warm,
    unconditionally supportive, and reassuring. No tips — just pure belief in the player.
    Bubble animation: bubbleCheer (same energetic bounce as tier 3).
    e.g. "You have been so persistent and I think that is genuinely wonderful! 🌈
         Please don't give up — the breakthrough is coming. I just know it!"

  ANIMATION DETAIL:
  bubbleIn (normal):     gentle 8px rise, 0.35s
  bubbleEncourage (T2):  6px rise → 4px overshoot → settle, 0.5s, softer bounce
  bubbleCheer (T3/T4):   8px rise → 7px overshoot → 3px dip → 3px bounce → settle,
                         0.65s, double-bounce communicates Hearty's physical excitement

  RESET CONDITIONS:
  - Player wins: lossStreak → 0 (Hearty responds with full celebration)
  - Draw: lossStreak → 0 (treated as a positive outcome for the player)
  - Mode switch (vs Hearty / 2P / Watch): lossStreak → 0
  - New match (startNewMatch): lossStreak → 0 via newGame() → setState

  SCOPE:
  Loss tracking only applies in vs-Hearty (ai) mode. 2-player and Watch modes are
  unaffected. Ultimate and Misère modes have their own separate state objects and
  are not tracked by this system (they use their own encouragement lines).

CHANGE 1: Page Visibility API — pause all animations and timers when app is backgrounded
  On mobile, when the user switches away from the app (lock screen, home screen,
  another app), the browser fires document.visibilitychange with document.hidden=true.
  Previously nothing responded to this, so the fireworks rAF loop, CSS animations,
  and the auto-tip interval all kept running in the background — wasting CPU and
  battery.

  WHAT WAS ADDED:

  _appVisible flag (boolean, default true):
    Single source of truth. Set by the visibilitychange listener.

  animateFireworks() — rAF loop refactored:
    Previously ran unconditionally on every frame forever, even with zero particles.
    Now:
    - Returns immediately (without rescheduling) when _appVisible is false.
    - Also stops rescheduling when the particle array is empty — the loop only
      runs while fireworks are actually visible. When a new burst is created,
      _startFireworksLoop() kicks it back off.
    - _rafId tracks the active requestAnimationFrame handle.

  _startFireworksLoop() helper:
    Starts the rAF loop only if not already running and app is visible.

  createFirework() wrapped:
    The original function is wrapped so _startFireworksLoop() is called
    automatically whenever a new burst is spawned.

  Auto-tip setInterval — guarded:
    Added `if(!_appVisible) return;` as the first line so Hearty's 15-second
    health tips are suppressed while the app is in the background.

  CSS animations — paused via class:
    Added body.app-hidden * { animation-play-state: paused !important; }
    This freezes every CSS keyframe animation on the page (Hearty breathing,
    title glow, floating hearts, thinking dots, etc.) with a single rule.

  visibilitychange event listener:
    On hide  → adds .app-hidden to body, suspends AudioContext
    On show  → removes .app-hidden, resumes AudioContext,
               restarts rAF loop if particles are queued

  BROWSER COMPATIBILITY:
    Page Visibility API is supported in all modern mobile browsers including
    Android WebView (used by Capacitor APK builds) since Android 4.4.
    Graceful degradation: if the API is absent, _appVisible stays true and
    all existing behaviour is unchanged.
